Node Explorer — Implementation Plan

A Norton Commander–style two-pane terminal file manager. Built on Node 24 native type stripping — no compiler, no build step.

Context

We're building Node Explorer, a Norton Commander–style two-pane terminal file manager, from scratch (the project directory is currently empty). It is inspired by disk-reclaim and deliberately reuses its tech stack and conventions. Where disk-reclaim is a read-only disk-usage viewer, Node Explorer is an interactive file manager: two independently-navigable panes, Tab to switch the active pane, per-file sizes, and full CRUD operations (copy, move, delete, rename, mkdir), plus opening files in vim and returning to the explorer.

Why this stack

  • Node with types stripped at runtime, no compiler. Node v24.18 (installed) runs .ts files directly via native type stripping — no tsc, no build step.
  • The critical constraint: native type stripping erases type annotations but cannot transform JSX. disk-reclaim sidesteps this by using htm (hyperscript tagged templates) instead of JSX with Ink. We do the same — this is the linchpin that lets us use Ink (React for terminals) in plain .ts files with zero build tooling.
Confirmed decisions
  • Keybindings: support both Norton Commander F-keys and letter shortcuts.
  • Directory sizes: show <DIR>; compute recursive size on demand (press s).
  • v1 operations: Copy, Move, Delete, Make directory, Rename, View (F3), Edit-in-vim (F4).

Tech Stack

  • Runtime: Node ≥ 24 (native type stripping, on by default).
  • Deps (match disk-reclaim): ink ^5react ^18htm ^3
  • No JSX, no tsc for running. Files are .ts, rendered with const html = htm.bind(React.createElement).
  • Optional dev-only: typescript as a devDependency for editor/CI type checking via tsc --noEmit — never used to build or run.

Key Design Details

1. htm + Ink, no JSX

Each component module: import React from 'react'; import htm from 'htm'; const html = htm.bind(React.createElement), then return html`<${Box} flexDirection="column">...`. Interpolate child components with <${Pane} .../>.

2. Type-stripping rules to follow (so Node runs it unmodified)

  • Local imports use the explicit .ts extension: import { listDir } from './core/fs.ts'.
  • Type-only imports use import type { Entry } from './types.ts'.
  • No enums, namespaces with runtime bodies, parameter properties, or declare fields (non-erasable syntax). tsconfig.json sets erasableSyntaxOnly: true + allowImportingTsExtensions: true + verbatimModuleSyntax: true to enforce this in the editor.

3. State (in App.ts)

  • Two PaneState objects: { cwd, entries, cursor, scrollOffset }.
  • active: 0 | 1; Tab toggles it. Each pane navigates independently.
  • modal state drives Dialog (none | confirm-delete | confirm-copy | confirm-move | input-mkdir | input-rename).
  • Entries sorted dirs first, then files, alphabetical, with a leading .. row (except at filesystem root) to navigate up. Enter on a dir → chdir; on a file → open in vim.

4. Input handling (useInput + fkeys.ts)

Ink's useInput(input, key) decodes tab, arrows, return, escape, delete, ctrl, etc., but not F-keys — those arrive as raw escape sequences in input (e.g. F5 = \x1b[15~). util/fkeys.ts maps those sequences to ids so we can bind F3–F10. Letter keys (v e c m n d r q s) are read directly from input. Both schemes call the same action handlers. When a modal is open, input routes to the Dialog instead of pane navigation.

5. Edit-in-vim (useEditor.ts) — the delicate part

const { setRawMode, isRawModeSupported, stdin } = useStdin()
// on F4/e:
if (!isRawModeSupported) return            // graceful no-op on non-TTY
setRawMode(false)                          // Ink's setRawMode, not process.stdin's
stdin.pause?.()
spawnSync('vim', [absPath], { stdio: 'inherit' })   // vim owns the TTY, we block
setRawMode(true)
stdin.resume?.()
// then refresh the active pane listing (file may have changed size)
Gotchas (from research): use Ink's setRawMode (via useStdin) so Ctrl+C stays handled; guard with isRawModeSupported; fully suspending input during the spawn prevents vim's terminal-background-color query response from leaking escape sequences into our UI. Editor is $EDITOR if set, else vim.

6. Core FS operations — reuse Node built-ins, don't hand-roll

OpImplementation
Listfs.readdir(dir, { withFileTypes: true }) + fs.lstat per entry for size/mtime
Copyfs.cp(src, dest, { recursive: true })
Movefs.rename, with cross-device fallback to cp + rm
Deletefs.rm(path, { recursive: true, force: true })
MkDirfs.mkdir
Renamefs.rename
  • Copy/Move default destination = the other pane's cwd (Norton Commander behavior), shown/confirmed in the Dialog. Delete/overwrite always confirm first.
  • core/dirSize.ts: recursive lstat walk summing file sizes; triggered by s, result cached on the entry and shown in place of <DIR>.

7. Viewport / scrolling (FileList.ts)

Visible rows = process.stdout.rows minus header/footer. Compute window start from cursor so the selection stays in view (clamp + keep-centered, like disk-reclaim's src/ui/App.js). Highlight the cursor row; dim the inactive pane's cursor.

Verification

  1. Run it: npm install then npm start (node src/index.ts). A two-pane explorer appears.
  2. Manual walkthrough (single terminal):
    • Tab switches the active pane; arrows move the cursor; Enter descends into a dir and .. goes up — each pane independent.
    • F7/n → mkdir dialog creates a folder; it appears in the list.
    • F5/c on a file → copy dialog (dest = other pane) → file appears in the other pane.
    • F6/m → move; F8/d → delete (with confirm); r → rename in place.
    • s on a <DIR> row → recursive size replaces <DIR>.
    • F3/v → in-app read-only viewer; Esc closes it.
    • F4/e on a file → vim opens full-screen; on :q control returns to Node Explorer at the same pane/cursor, and the file's size refreshes. (Primary acceptance test.)
    • F10/q quits cleanly (terminal restored, raw mode off).
  3. Unit tests: node --test covers pure/core logic — format.ts (size/date formatting), fs.ts sorting, and ops.ts against a temp dir (mkdir/copy/rename/delete round-trips).
  4. Type check (optional, dev): npx tsc --noEmit reports zero errors and confirms all syntax is erasable (so Node runs the sources untouched).

Out of Scope for v1 (possible follow-ups)

Multi-file marking (Insert key), archive browsing, built-in text editor, filesystem watching/auto-refresh, config file for custom keybindings.
